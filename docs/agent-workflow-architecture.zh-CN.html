SurfSense Agent 工程说明（HTML 版）

覆盖当前实现、目标技术路线、工作流程图与代表性数据处理案例。

版本 v1.0 · 日期 2026-03-11 · 适用范围 surfsense_backend + surfsense_web

1. 项目目标

多源检索 RAG 流式输出 可扩展 Agent
  1. 从个人知识源与外部源检索高相关证据。
  2. 通过混合检索与重排提升回答质量，并输出 citation。
  3. 流式返回 terminal、sources、正文增量、后续问题。
  4. 向意图识别、规划、反思、Tool/Skill/Memory 架构演进。

能力拆解

  1. 意图识别：输出 intent/confidence/needs_web/needs_memory。
  2. 规划：把任务拆解为可执行步骤，并记录 execution trace。
  3. Tool + Skill：将现有检索与外部搜索统一抽象为可调度能力。
  4. 反思：覆盖率、引用完整性、冲突处理通过后再对外输出。
  5. Memory：会话记忆 + 长期记忆分层管理，并支持有条件写回。

3.5 Researcher 具体架构拆解

组件分层（按代码职责）

请求入口层

chats_routes 负责鉴权、参数校验、流式响应入口。

任务封装层

stream_connector_search_results 负责把请求映射为 graph config 与 state。

主编排层

Researcher 图：改写查询 -> QnA 工作流 -> 后续问题。

子编排层

QnA 图：rerank_documents -> answer_question。

检索聚合层

ConnectorService 统一调度本地索引源与外部 API 源。

检索算子层

Chunk/Document Hybrid Search（向量 + 全文 + RRF）。

质量增强层

Reranker + token 预算裁剪，提升相关性与稳定性。

流式协议层

StreamingService 输出 terminal/sources/text/further_questions。

节点 I/O 契约

  1. reformulate_user_query：输入 user_query/chat_history，输出 reformulated_query。
  2. handle_qna_workflow：输入 connectors/documents/query，输出 final_report 与 reranked_documents。
  3. generate_further_questions：输入 chat_history/reranked_documents，输出 follow-up 问题数组。
  4. QnA.rerank_documents：无 reranker 时自动降级为原顺序。
  5. QnA.answer_question：拼接 citation/custom prompt，并执行 token 预算裁剪。

状态模型与当前边界

  1. 当前状态核心字段：db_session、chat_history、reformulated_query、reranked_documents、final_answer。
  2. 当前记忆能力以会话短期记忆为主，长期用户记忆尚未落库与召回。
  3. 当前编排是固定线性链路，尚未进入动态规划与反思回路。

失败与降级策略

  1. 单连接器失败不阻断整体流程，继续检索其他源。
  2. 无文档时降级到 no-documents prompt，保证可回答。
  3. 无 reranker 自动降级为原始排序。
  4. token 超预算时做文档裁剪，避免超窗失败。
  5. 后续问题 JSON 解析失败时返回空数组，不影响主回答。

4. 数据处理闭环

4.1 入库链路

  1. 连接器触发异步索引任务（Celery）。
  2. 源数据标准化，生成 unique_identifier_hash/content_hash。
  3. 文档摘要、分块、向量化后写入 Document/Chunk。

4.2 查询链路

  1. 接收 query 与检索配置。
  2. 改写查询并跨源检索。
  3. RRF 融合 + 去重 + rerank。
  4. 上下文预算控制后生成答案与引用。
  5. 流式输出 terminal、sources、text、further questions。
RRF 评分形式：score(d) = 1/(k + r_semantic(d)) + 1/(k + r_keyword(d))

5. 代表性案例（逐步数据处理）

案例 A：发布复盘（Slack + Jira + Notion）

  1. 输入：发布阻塞问题 + 指定三类连接器。
  2. 处理：改写查询 -> 多源检索 -> 时间与语义聚合。
  3. 输出：阻塞点、根因、修复动作与 citation。

案例 B：冲突核验（GitHub vs Gmail）

  1. 输入：PR 已修复但用户仍报错。
  2. 处理：提取提交时间、邮件时间并做时间轴对齐。
  3. 输出：冲突解释与下一步排查建议。

案例 C：知识缺口（本地命中不足）

  1. 输入：本地库无足够证据的问题。
  2. 处理：To-Be 路线下自动触发外部检索 skill 补证据。
  3. 输出：分层置信答案，并写回“知识缺口记忆”。

6. 分阶段落地

  1. Phase 1：先加 detect_intent + planner，不动底层检索实现。
  2. Phase 2：接入 reflector，形成“检索-生成-反思-回补”闭环。
  3. Phase 3：接入长期记忆与 skill registry，形成统一能力总线。
