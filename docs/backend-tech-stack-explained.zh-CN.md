# SurfSense 后端技术栈通俗讲解

这篇文档的目标不是解释每个框架的官方定义，而是用尽量通俗的例子，帮助你理解 SurfSense 后端里各个部件分别负责什么，以及它们是怎样协同工作的。

可以把 SurfSense 后端想成一家“会收资料、会整理资料、会回答问题”的研究所。用户上传文件、连接外部数据源、发起提问时，后端会把任务分给不同岗位来完成。

## 先看全局：它们像一条流水线

最简单的理解方式是：

- FastAPI 负责接请求
- FastAPI Users 负责登录和身份校验
- Celery 和 Redis 负责处理耗时任务
- Chonkie 负责把长文档切成小块
- 向量嵌入和 pgvector 负责按语义找内容
- PostgreSQL 负责存储业务数据
- SQLAlchemy 负责用 Python 管理数据库数据
- LangGraph 负责安排 AI 工作流步骤
- LangChain 和 LiteLLM 负责把大模型真正接进来
- Rerankers 负责把检索结果重新排得更准

如果把它翻译成人话，就是：

“前台接单，后台加工，档案室存档，检索系统找资料，AI 助手组织答案。”

---

## 1. FastAPI：后端的前台接待

FastAPI 像研究所的前台。

前端页面每发来一个请求，比如“上传文件”“开始聊天”“查看文档”“用户登录”，都先到 FastAPI 这里。FastAPI 判断这个请求应该交给谁处理，再把结果返回给前端。

### 通俗例子

你去图书馆前台说：“我要办借书证。”

前台不会亲自去印卡、建档、查权限，但它知道：

- 这个需求该走哪个窗口
- 需要你提供哪些材料
- 办完之后结果该怎么回给你

FastAPI 做的就是这种“统一接待和分发”的工作。

### 在 SurfSense 里它常干什么

- 提供上传、搜索、聊天、登录等 API
- 校验请求参数
- 调用数据库、认证、AI、任务队列等后端模块
- 把结果流式返回给前端

---

## 2. PostgreSQL with pgvector：既是档案室，也是语义检索柜

PostgreSQL 是总档案室，pgvector 是档案室里新增的“语义检索功能”。

普通数据库擅长回答这类问题：

- 这篇文档是谁上传的？
- 它属于哪个搜索空间？
- 这条聊天记录是什么时候产生的？

但用户问的问题往往是：

“有没有讲过 Celery 和异步任务的内容？”

这时就不只是按标题或关键词查了，还要按“意思相近”去找。pgvector 就是为这个能力服务的。

### 通俗例子

普通档案柜只能按编号找文件。

pgvector 则让你可以说：

“帮我找几份内容和这句话最像的资料。”

### 在 SurfSense 里它常干什么

- 存用户、文档、聊天、权限等结构化数据
- 存文档切块后的向量
- 支持语义相似度搜索

---

## 3. SQLAlchemy：程序员和数据库之间的翻译员

SQLAlchemy 像数据库管理员使用的一套标准表格系统。

开发者不必每次都手写 SQL，很多时候只需要用 Python 对象表达“这是一个用户”“这是一个文档”“这是一个文档片段”，SQLAlchemy 会帮忙把这些对象映射成数据库操作。

### 通俗例子

你不是直接冲进档案室自己改文件柜，而是填写标准申请表：

- 新建一份文档记录
- 查找一个用户
- 更新一条聊天内容

SQLAlchemy 就像这套申请表和翻译流程。

### 在 SurfSense 里它常干什么

- 定义数据库模型
- 操作 PostgreSQL 中的数据
- 管理 Python 对象和数据表之间的关系

---

## 4. Alembic：数据库结构的施工记录员

项目不断开发时，数据库表结构会变，比如新增字段、增加表、修改索引。Alembic 的作用就是把这些变化记成一条条可执行的迁移记录。

### 通俗例子

图书馆以前只记录一本书的标题和作者。

后来决定每本书还要记录“所属主题”。这就不是改一份纸质登记表那么简单，而是整个档案系统都要统一升级。Alembic 就是负责记录和执行这次升级的人。

### 在 SurfSense 里它常干什么

- 管理数据库结构变更
- 保证不同环境的数据库结构一致
- 让团队知道每次数据库改了什么

---

## 5. FastAPI Users：用户系统和门禁系统

FastAPI Users 像研究所的门禁和账户中心。

它负责注册、登录、JWT 认证、OAuth 登录这类通用账号能力。这样项目就不用从零自己造一整套登录系统。

### 通俗例子

研究所不是谁都能进。有的人只能看公开区，有的人能进入资料室，有的人还能管理团队成员。

FastAPI Users 做的事情就是：

- 确认你是谁
- 确认你有没有权限
- 给你发一张能证明身份的通行证

### 在 SurfSense 里它常干什么

- 用户注册和登录
- JWT 身份认证
- OAuth 登录支持
- 账户相关基础功能

---

## 6. LangGraph：AI 工作流的导演

LangGraph 像导演，负责安排 AI 代理“先做什么、后做什么”。

真实的问答流程通常不是只把用户问题直接丢给模型就结束。更合理的流程可能是：

1. 先改写问题
2. 再去检索资料
3. 再组织上下文
4. 再生成答案
5. 最后给出可追问的问题

LangGraph 的价值就在于把这些步骤编排成稳定可控的流程。

### 通俗例子

导演不会让演员直接上台自由发挥，而是安排流程：

- 先采访
- 再调档
- 再整理素材
- 最后对外发布

LangGraph 就是这个导演。

### 在 SurfSense 里它常干什么

- 编排多步问答流程
- 传递各步骤之间的状态
- 让 AI 回答过程更稳定、更可调试

---

## 7. LangChain：AI 应用的工具箱

LangChain 更像一个工具箱。

它提供了很多通用组件，帮助程序组织提示词、调用模型、处理消息、拼接上下文。LangGraph 负责“流程怎么走”，LangChain 负责“流程里的工具怎么用”。

### 通俗例子

如果 LangGraph 是导演，LangChain 就像：

- 摄影机
- 提词器
- 采访模板
- 素材整理板

### 在 SurfSense 里它常干什么

- 封装模型调用
- 处理消息和上下文
- 支持检索增强问答流程

---

## 8. LiteLLM：不同大模型的万能转接头

LiteLLM 的核心作用，是让项目更容易接不同厂商的大模型。

不同模型服务商的接口风格往往不完全一样。LiteLLM 会把这些差异尽量抹平，让项目用更统一的方式接入多种模型。

### 通俗例子

你家里有很多不同国家的插头。LiteLLM 就像一个万能转换插座。

你不用为了每个电器都重装墙上的电源接口。

### 在 SurfSense 里它常干什么

- 统一对接不同 LLM 提供商
- 方便切换模型
- 支持不同任务使用不同模型

例如：

- 快速回答用响应更快的模型
- 长文总结用上下文更长的模型
- 某些推理步骤用更擅长分析的模型

---

## 9. Rerankers：检索结果的二次筛选编辑

第一次检索出来的内容，不一定排序就是最好的。

Reranker 的工作是把候选结果重新看一遍，把最相关的放在更前面。

### 通俗例子

实习生先找到了 20 份可能相关的资料，但顺序不太准。

资深编辑再过一遍，把真正最相关的 5 份排到最前面。Reranker 就是在做这个工作。

### 在 SurfSense 里它常干什么

- 对候选检索结果重新排序
- 提高上下文质量
- 让最终回答更贴题

---

## 10. 混合搜索：关键词搜索和语义搜索一起上

混合搜索的思路很朴素：不要只押宝一种搜索方式。

有时候用户的问题里关键词非常明确，全文搜索很有效。也有时候用户表达得很自然，措辞和原文不同，这时语义搜索更有效。混合搜索就是两种都用，再把结果融合起来。

### 通俗例子

你找资料时，一边翻目录找关键词，一边问图书管理员：

“有没有意思差不多的内容？”

两边结果合起来，通常比单独用一种方法更靠谱。

### 在 SurfSense 里它常干什么

- 同时进行全文检索和向量检索
- 提高召回率
- 降低漏掉相关内容的概率

---

## 11. 向量嵌入：把文字变成机器能比较的坐标

向量嵌入可以理解为：把一段文字转成一串数字坐标。

这些数字不是给人看的，而是给机器比较“语义距离”用的。意思越接近，两段文字在向量空间里的距离通常越近。

### 通俗例子

想象有一张很大的地图：

- 谈数据库的内容在一个区域
- 谈异步任务的内容在另一个区域
- 谈权限系统的内容在另一个区域

每段文本都会落在地图上的某个位置。用户提问后，系统也会把问题放上地图，再去找附近的内容。

### 在 SurfSense 里它常干什么

- 把文档块转成向量
- 把用户问题转成向量
- 支持语义搜索

---

## 12. pgvector：让数据库学会比较“语义距离”

pgvector 是 PostgreSQL 的向量扩展。

没有它，数据库更擅长按字段精确匹配。加上它之后，数据库就能直接存储向量并计算“哪些内容更相似”。

### 通俗例子

原本档案室只能按编号找资料。装上 pgvector 之后，档案室开始能回答：

“哪几份资料和这段话的意思最接近？”

### 在 SurfSense 里它常干什么

- 存储嵌入向量
- 执行向量相似度搜索
- 支撑语义检索能力

---

## 13. Redis：高速中转站和任务快递柜

Redis 在这个项目里，最容易理解成“任务中转站”。

有些工作不适合在用户请求的当下慢慢做，比如大文件解析、切块、向量化、摘要生成。这些任务会被先放到队列里，后台再慢慢处理。Redis 就负责做这个中间层。

### 通俗例子

前台接到一份资料处理申请，不会立刻在柜台旁边开始拆文档、切分和归档，而是先把任务单放进一个快递柜。后台员工再从柜子里取任务处理。

Redis 就像这个快递柜。

### 在 SurfSense 里它常干什么

- 作为 Celery 的消息代理
- 协助异步任务流转
- 提供高性能中间存储

---

## 14. Celery：后台干活的工人队伍

Celery 是用来处理异步任务的。

如果上传完文档后，前端必须一直等到“文本提取完成、切块完成、向量生成完成、数据库写入完成”才返回，用户体验会很差。Celery 的作用，就是把这些重活挪到后台。

### 通俗例子

你把一整箱资料交给研究所，前台给你一个“已受理”的回执，然后后台工作人员开始慢慢整理。你不需要站在前台等他们全部处理完。

Celery 就是后台那群真正干活的人。

### 在 SurfSense 里它常干什么

- 异步处理文档上传后的任务
- 执行耗时的数据处理流程
- 减少请求阻塞

---

## 15. Flower：Celery 的监控大屏

Flower 不是核心业务逻辑，而是监控工具。

它可以让开发者看到：

- 哪些 Celery 任务正在跑
- 哪些成功了
- 哪些失败了
- 队列是不是堆积了

### 通俗例子

车间主管抬头看生产监控屏，就知道哪条流水线卡住了。

Flower 就是这块监控屏。

### 在 SurfSense 里它常干什么

- 观察异步任务运行状态
- 辅助排查任务问题
- 查看任务队列是否健康

---

## 16. Chonkie：文档切分师

长文档通常不能直接整篇拿去做检索或问答。

原因很简单：

- 太长
- 不利于精确命中
- 成本高
- 容易把无关内容一起带进去

Chonkie 的作用，就是把长文档切成更适合处理的小块，再去做嵌入和存储。

### 通俗例子

你不会把一本 300 页的书整本交给助手说“帮我找答案”，你更可能把它拆成章节、段落、知识卡片。这样后面查找时才快。

Chonkie 做的就是“拆书成卡片”的工作。

### 在 SurfSense 里它常干什么

- 把长文档切块
- 让每个块更适合做向量嵌入
- 提高后续检索和回答质量

其中常见能力包括：

- AutoEmbeddings：更灵活地选择嵌入模型
- LateChunker：根据嵌入模型可处理长度，优化切分方式

## 一个完整例子：用户上传文档并提问时，后端怎么协同工作？

下面用一个完整场景把这些组件串起来。

### 场景

用户上传一份“产品技术方案.pdf”，然后提问：

“这份方案里有没有提到异步任务和通知机制？”

### 第一步：接收上传

FastAPI 接到上传请求。

它会做的事情包括：

- 接收文件
- 校验请求
- 确认用户身份和权限
- 创建对应的处理任务

此时 FastAPI 更像前台，它先完成受理。

### 第二步：把重活交给后台

Celery 接过任务，Redis 负责把任务消息传给后台 worker。

这一步的核心思想是：

“前台先受理，后台再慢慢做。”

所以用户不需要一直卡在上传页面等待所有处理完成。

### 第三步：提取内容并切块

后台开始处理文档内容：

- 提取文本
- 清洗内容
- 用 Chonkie 切分成多个文档块

这些块就像一张张知识卡片，而不是一本整书。

### 第四步：生成向量并存库

每个文档块会做向量嵌入，然后存入 PostgreSQL。

其中：

- SQLAlchemy 负责用 Python 方式组织这些数据
- PostgreSQL 负责保存业务数据
- pgvector 负责保存和检索向量

这一步完成后，系统就具备“按语义找内容”的能力了。

### 第五步：用户提问

用户发起聊天请求时：

- FastAPI 接到问题
- FastAPI Users 确认用户身份
- LangGraph 开始驱动整条问答流程

### 第六步：先检索，不直接瞎答

系统会先找资料，而不是直接让模型凭空生成。

这时会发生几件事：

- 用户问题被转成向量
- pgvector 去找语义最接近的文档块
- 全文搜索再按关键词找一遍
- 混合搜索把两类结果融合
- Reranker 再把顺序调得更准

这样做的好处是，模型回答时有依据，不容易胡说。

### 第七步：组织答案并返回前端

LangGraph 负责安排步骤，LangChain 负责处理消息与上下文，LiteLLM 负责接入真正的大模型。

最后模型根据检索到的资料生成答案，FastAPI 再把答案返回给前端。

如果开启流式输出，用户会看到答案一点点出现，而不是等全部生成完才看到。

---

## 一句话总结

SurfSense 后端不是“一个框架做所有事”，而是多个组件分工合作：

- FastAPI 负责对外接请求
- FastAPI Users 负责身份认证
- Celery + Redis 负责后台异步处理
- Chonkie 负责切文档
- 向量嵌入 + pgvector 负责语义检索
- PostgreSQL + SQLAlchemy 负责数据存储和管理
- LangGraph + LangChain + LiteLLM 负责 AI 问答流程
- Rerankers 负责把资料排得更准

最终这些部件协同起来，才让 SurfSense 既能“收资料”，又能“理解资料”，还能“基于资料回答问题”。
